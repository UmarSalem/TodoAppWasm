TodoAppWasm Auth and JWT Guide

Junior developer notes for tasks 034_backend_auth_user_accounts and 035_backend_jwt_authorization.

Short idea: task 034 teaches the backend how to register and login users safely. Task 035 adds JWT tokens so the backend can protect Todo endpoints and know which user is making each request.

1. Why We Added Authentication

Before these tasks, the API could create users and todos, but it did not know who was logged in. A client could send an ownerId in JSON, and the backend trusted it.

That is risky because a user could change JSON and create or edit todos for another user. Authentication fixes this by letting the server identify the caller from a signed token.

2. Task 034: User Accounts and Password Hashing

The goal of task 034 was to let users register with a password and login with that password.

User Model

We added a password hash to Shared/Models/User.cs.

public string PasswordHash { get; set; } = string.Empty;

We do not store the original password. We store only a hash. If someone opens the database, they should not see real user passwords.

User Creation DTO

We changed UserCreationDto so registration receives username and password.

public string UserName { get; set; }
public string Password { get; set; }

The DTO is the shape of JSON coming from Swagger or the frontend.

Password Hasher

We added Pbkdf2PasswordHasher. Its job is to turn a password into a secure stored value.

public string Hash(string password)
{
    byte[] salt = RandomNumberGenerator.GetBytes(SaltSize);
    byte[] key = DeriveKey(password, salt, Iterations);

    return string.Join(
        '$',
        Algorithm,
        Iterations,
        Convert.ToBase64String(salt),
        Convert.ToBase64String(key));
}

The salt makes hashes unique. Two users with the same password still get different hash values.

Password Verification

public bool Verify(string password, string passwordHash)
{
    ...
    return CryptographicOperations.FixedTimeEquals(actualKey, expectedKey);
}

When the user logs in, we do not decrypt anything. We hash the entered password again and compare it with the saved hash.

Important: password hashing is one-way. We should never be able to recover the original password from the database.

User Logic

In UserLogic, new users are saved with a hash and the default role.

User toCreate = new User
{
    UserName = usertoCreate.UserName,
    PasswordHash = passwordHasher.Hash(usertoCreate.Password),
    Role = UserRoles.User
};

New users become normal User accounts. They cannot register themselves as Admin.

Login Logic

if (existing == null || !passwordHasher.Verify(dto.Password, existing.PasswordHash))
{
    throw new AppValidationException("Invalid username or password.");
}

The error message is intentionally general. We do not reveal whether the username or password was wrong.

Database Migration

We added a migration to add PasswordHash to the Users table.

dotnet ef database update --project EfcDataAccess --startup-project WebAPI

Without applying this migration, Swagger can return 500 because the code tries to save a column that the database does not have yet.

3. Task 035: JWT Authorization

The goal of task 035 was to protect Todo endpoints. After login, the user receives a JWT token. Swagger or the frontend sends that token when calling protected endpoints.

Login Response

Login now returns a DTO with the token, safe user info, and expiry time.

public string Token { get; set; }
public UserReadDto User { get; set; }
public DateTime ExpiresAtUtc { get; set; }

JWT Claims

Claims are trusted identity data inside the signed token.

Claim[] claims =
[
    new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
    new Claim(ClaimTypes.Name, user.UserName),
    new Claim(ClaimTypes.Role, user.Role)
];
Claim Meaning
NameIdentifier The logged-in user's database id.
Name The logged-in user's username.
Role The user's role, for example User or Admin.

JWT Configuration

In Program.cs, the API validates tokens with issuer, audience, expiry time, and signing key.

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer = builder.Configuration["Jwt:Issuer"],
            ValidAudience = builder.Configuration["Jwt:Audience"],
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwtKey))
        };
    });

The middleware order matters:

app.UseAuthentication();
app.UseAuthorization();

UseAuthentication() reads the token and creates User.Claims. UseAuthorization() checks whether the endpoint can run.

Swagger Authorize Button

Swagger needs a Bearer security definition before it shows the Authorize button.

options.AddSecurityDefinition("Bearer", new OpenApiSecurityScheme
{
    Name = "Authorization",
    Type = SecuritySchemeType.Http,
    Scheme = "bearer",
    BearerFormat = "JWT",
    In = ParameterLocation.Header
});

After login, paste the token into Swagger as:

Bearer eyJhbGciOiJIUzI1NiIs...

4. Roles vs Ownership

This is one of the most important ideas in the project.

Concept Example Meaning
Role User, Admin What type of account the logged-in person has.
Ownership todo.OwnerId == loggedInUserId Whether this exact todo belongs to the logged-in user.
Clean rule: Admin and User are roles. Owner is not a role. Owner is checked by comparing a todo's OwnerId with the user id from the JWT token.

5. Protected Todo Endpoints

We added [Authorize] to TodosController.

[Authorize]
public class TodosController : ControllerBase

This means the user must login and send a valid token before calling Todo endpoints.

Create Todo

int userId = GetCurrentUserId();
TodoCreationDto secureDto = new(userId, dto.Title, dto.Description);

The API no longer trusts ownerId from JSON. It uses the user id from the JWT token.

List Todos

SearchTodoParametersDto parameters = isAdmin
    ? new(userName, userId, completedStatus, titleContains, descriptionContains)
    : new(null, currentUserId, completedStatus, titleContains, descriptionContains);

Admins can inspect broader data. Normal users only see their own todos.

Access Check

if (User.IsInRole(UserRoles.Admin) || todo.OwnerId == GetCurrentUserId())
{
    return;
}

throw new NotFoundException($"Todo with id {todo.Id} was not found.");

If a normal user tries another user's todo id, the API returns 404. This avoids revealing that the other user's todo exists.

6. How to Test in Swagger

  1. Run migrations so the database has PasswordHash and Role.
  2. Create a user with POST /Users.
  3. Login with POST /Users/login.
  4. Copy the token from the login response.
  5. Click Swagger's Authorize button.
  6. Paste Bearer {token}.
  7. Call Todo endpoints.

7. Common Errors

Error Likely reason Fix
500 when creating user Database is missing PasswordHash or Role. Run dotnet ef database update --project EfcDataAccess --startup-project WebAPI.
No Authorize button in Swagger Swagger Bearer setup is missing or API was not restarted. Restart WebAPI and hard refresh Swagger.
401 Unauthorized Missing, expired, or invalid token. Login again and paste Bearer {token} into Swagger.
404 for an existing todo The todo belongs to another user. Use the owning user's token, or login as Admin.

8. Final Mental Model

Register:
  username + password
  -> hash password
  -> save UserName, PasswordHash, Role

Login:
  username + password
  -> verify hash
  -> return JWT

Todo request:
  frontend sends JWT
  -> backend validates token
  -> backend reads user id and role from claims
  -> backend allows or blocks the action
